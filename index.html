Purpose

What Is This?

A structured foundation so AI sessions stay coherent and grounded across multiple conversations.

AI coding assistants are powerful, but they reset between sessions. Without a persistent context system, every conversation starts from scratch — you re-explain the architecture, re-establish constraints, and re-discover past decisions.

White Hat Label gives every project a documentation framework that AI sessions can load at the start of each conversation. Constraints, decisions, and session history are preserved — not lost.

Objectives

  • ✦ Eliminate the cost of re-establishing context at the start of every AI session
  • ✦ Enforce quality guardrails that survive context resets
  • ✦ Record architectural decisions, feedback loops, and breakthroughs so they are never lost
  • ✦ Give any developer — or AI model — a clear reading order to get up to speed immediately
  • ✦ Provide a reusable starting point for any HTML/CSS/JS or React/Next.js project

Self-Correcting

Reflective Architecture

The system is designed to catch drift between documentation and code before it compounds.

Reflective Sync

Structured prompts to run at session start, mid-session, and end — catches drift between docs and code before it accumulates. Defined in docs/architecture/REFLECTIVE-SYNC.md.

Contradiction Hunt

Feed all L1/L2 architecture docs to a second model (Gemini or similar). Ask for the top five internal contradictions. Surface ambiguities that a single session wouldn't notice. See six-hats.md §Contradiction Hunt.

Recursive Architecture Test

Feed ARCHITECTURE.md and SYSTEM.md to a fresh model. Ask it to describe the project. Compare its response to the actual code — divergences = documentation debt. See GEMINI-CONSULTANCY.md Pattern 5.

Codebase Audit

Periodic full audit against G1–G13. The file CODEBASE-AUDIT.md chunks the codebase into manageable batches so the audit can be done across multiple sessions without losing track of what's been checked.

Breakthroughs Log

Root causes of past hard problems are recorded in BREAKTHROUGHS.md. Avoids re-diagnosing known issues — each entry includes what the problem was, what was tried, what worked, and why.

State Persistence

Session Plan System

Context resets don't cause lost work. Every session closes with a handoff; every session opens by reading it.

docs/plan/plan-rules.md
8 Operating Rules

When to write handoffs, how to mark tasks, how to record OPEN questions vs silent assumptions, and when to write mid-session checkpoints.

docs/plan/tasklist.md
Task Register

Canonical task list — every task (open and completed) lives here. Tasks are never deleted; the history is the record.

docs/plan/handoff_template.md
Handoff Template

Written at session end. Covers what was done, current state (working / not working), open questions, and the next session entry prompt.

Three Session Styles

Style When What to Load
Hybrid Default Day-to-day development, feature work, bug fixes Handoff + ARCHITECTURE.md + CORE_PATTERNS.md
Fresh / Red Team Architecture decisions, avoiding tunnel vision ARCHITECTURE.md + task description only — no handoff
Deep / Bloated Mystery bugs, >2 failed attempts Everything: handoff + ARCHITECTURE_EXTENSION.md + console output

Setup

Get Started in Four Steps

The documentation scaffold is ready. You replace the placeholders, then build.

  1. 1
    Clone the repo

    Clone or download white-hat-label. This gives you the full documentation scaffold with all 13 architecture docs and the plan system in place.

  2. 2
    Replace the placeholders

    Find all [PROJECT_NAME] markers in CLAUDE.md, docs/ARCHITECTURE.md, and docs/SYSTEM.md. Fill in the project name, domain, stack choices, and design tokens.

  3. 3
    Start a Claude Code session

    Use the entry prompt from docs/architecture/CLAUDE_ARCHITECTURE.md. Claude reads the architecture docs first, then picks up the task — constraints and conventions already in context.

  4. 4
    Build and close with a handoff

    At session end, Claude writes a handoff to docs/plan/handoff_YYYY-MM-DD.md. The next session opens by reading it — context is preserved, work continues.

Entry Prompt — Hybrid Session (Default)

Read docs/plan/handoff_[latest].md, docs/plan/tasklist.md, and docs/ARCHITECTURE.md.
The task is: [task description].
